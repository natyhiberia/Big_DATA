Sketch Your UI
⏱ 10 min
1

Before opening Claude Code, draw your app on paper or a whiteboard. This step is intentional — Claude Code generates better layouts when you describe something concrete. A rough sketch beats a paragraph of abstract requirements every time.

Your app layout — draw something like this:
┌──────────────────────────────────────┐ │ GDM Yield Predictor │ ├───────────────┬──────────────────────┤ │ Crop Type │ [dropdown] │ │ Country │ [dropdown] │ │ Rainfall (mm) │ [slider 200–2000] │ │ Temperature°C │ [slider 5–30] │ │ │ [PREDICT button] │ ├───────────────┴──────────────────────┤ │ Predicted Yield: 45,320 hg/ha │ │ = 4.5 tonnes/hectare │ ├──────────────────────────────────────┤ │ [Bar: my prediction vs country avg │ │ vs global avg] │ ├──────────────────────────────────────┤ │ [Feature importance chart] │ └──────────────────────────────────────┘
Sketch tips
  • Identify each input (dropdowns, sliders, buttons)
  • Identify each output (metric card, charts, table)
  • Note the chart types — bar chart? scatter? horizontal bar?
  • Once drawn, you can take a photo and attach it directly to Claude Code
Why sketch first?
Claude Code accepts photo attachments. A photo of your whiteboard sketch gives it spatial context that's impossible to convey in words alone. You'll get cleaner layouts with fewer revision rounds.
2
Create Project & Configure Claude Code
⏱ 15 min
2

We configure everything at the project level — the folder becomes a self-contained environment. Databricks FMAPI serves as the LLM for Claude Code so all AI usage bills to the workspace, not to individual Anthropic accounts.

A — Create the project folder
Mac / Linux & Windows PowerShell
# Create your app folder — all config lives inside it mkdir gdm-yield-app cd gdm-yield-app
B — Authenticate Databricks CLI with OAuth

Use databricks auth login instead of a PAT token — this opens a browser for OAuth login and saves a named profile you can reuse.

Mac / Linux & Windows PowerShell
# OAuth login — opens your browser to authenticate databricks auth login --host https://<workspace>.azuredatabricks.net # When prompted for a profile name, type something memorable: # Profile name: gdm-workshop # Verify the profile works databricks current-user me --profile gdm-workshop
Expected flow
  • Browser opens → log in with your Databricks workspace credentials (SSO)
  • Terminal prompts: Profile name (default): → type gdm-workshop and press Enter
  • current-user me returns your email — no errors
  • Profile saved at ~/.databrickscfg under [gdm-workshop]
Why OAuth instead of a PAT?
  • No token to copy, store, or rotate — uses your workspace SSO credentials
  • The named profile (gdm-workshop) can be referenced by both the CLI and the AI Dev Kit
  • Token is short-lived and auto-refreshed; PATs are long-lived and easy to leak
C — Configure Claude Code to use Databricks FMAPI

Create a .claude/settings.json file inside the project folder. This is a project-level file — it overrides the global Claude Code config only when you run claude from this directory.

Mac / Linux — create the config directory
mkdir -p .claude
Windows PowerShell
New-Item -ItemType Directory -Force .claude

Now create gdm-yield-app/.claude/settings.json with this content:

📄 .claude/settings.json — paste and save
{ "env": { "ANTHROPIC_BASE_URL": "https://<workspace>.azuredatabricks.net/serving-endpoints/databricks-claude-sonnet-4-5", "DATABRICKS_CONFIG_PROFILE": "gdm-workshop" } }
Replace the placeholders
  • <workspace> — your workspace subdomain (e.g. adb-1234567890.12)
  • databricks-claude-sonnet-4-5 — the Claude model endpoint name in your workspace. Ask the facilitator for the exact name.
  • gdm-workshop — the profile name you chose in step B. Change if you used a different name.
How this works
  • Claude Code reads .claude/settings.json on startup when launched from this folder
  • ANTHROPIC_BASE_URL redirects LLM calls to your Databricks workspace endpoint
  • DATABRICKS_CONFIG_PROFILE tells the Databricks SDK to use the OAuth token from the gdm-workshop profile — no PAT needed
  • All AI usage routes through Databricks and is governed by your workspace policies
D — Install AI Dev Kit (adds MCP + Skills to Claude Code)
Mac / Linux
# Run from inside gdm-yield-app/ — uses the gdm-workshop profile automatically bash <(curl -sL https://raw.githubusercontent.com/databricks-solutions/ai-dev-kit/main/install.sh) # Select: Claude Code → app-developer profile → confirm workspace URL
Windows PowerShell
# Run from inside gdm-yield-app\ irm https://raw.githubusercontent.com/databricks-solutions/ai-dev-kit/main/install.ps1 | iex # Select: Claude Code → app-developer profile → confirm workspace URL
E — Launch Claude Code
# Make sure you are inside gdm-yield-app/ before running claude
Expected result
  • Claude Code opens with the > prompt
  • LLM calls route to your Databricks workspace (check with /model inside Claude Code)
  • AI Dev Kit MCP tools are available — Claude Code can query your Delta tables, deploy apps, etc.
  • Project folder contents so far: .claude/settings.json — Claude Code will add everything else
Project folder structure after this step:
gdm-yield-app/ └── .claude/ ├── settings.json ← Databricks FMAPI + project permissions └── (AI Dev Kit adds MCP config and skills here)

Describe Your App to Claude Code
⏱ 50 min
3

Inside the Claude Code session, paste the master prompt below — replacing the bracketed values with your own endpoint URL and username from Module 1. Claude Code will generate app.py, requirements.txt, and any helper files.

Master prompt — paste into Claude Code
Build a Databricks App for GDM crop yield prediction. CONTEXT: - Model Serving endpoint URL: <paste your endpoint URL from Module 1 Step 6> - Delta table for reference data: <CATALOG>.<USERNAME>.yield_trials (replace CATALOG and USERNAME with your actual values from Module 1) - Table columns: Area (country), Item (crop type), Year, hg/ha_yield, average_rain_fall_mm_per_year, pesticides_tonnes, avg_temp IMPORTANT — encoding: The model is a sklearn Pipeline that includes OrdinalEncoder internally. The serving endpoint accepts RAW STRINGS — pass "Maize" and "Argentina" directly. Do NOT encode crop type or country before calling the endpoint. The payload fields are: Item, Area, average_rain_fall_mm_per_year, pesticides_tonnes, avg_temp, Year UI — use Gradio: - Title: "GDM Yield Predictor" - Input form: * Crop type dropdown (unique values of "Item" from the Delta table, sorted) * Country dropdown (unique values of "Area" from the Delta table, sorted) * Rainfall mm/year slider: range 200–2000, default 750 * Average temperature °C slider: range 5–30, default 18 * "Predict Yield" button Outputs shown after clicking Predict: 1. Large metric display: "Predicted Yield: X,XXX hg/ha" Also show the equivalent in tonnes/hectare (divide by 10,000) 2. Plotly bar chart (3 bars): - "My Prediction" (the model output) - "Country Average" (avg hg/ha_yield for selected country from Delta table) - "Global Average" (overall avg hg/ha_yield from Delta table) 3. A dataframe showing the last 5 predictions with inputs + outputs Backend requirements: - Pass crop/country names as raw strings — the endpoint pipeline handles encoding internally - Authenticate using WorkspaceClient() with NO arguments — auth is automatic both locally and in Databricks Apps - Call the Model Serving endpoint using requests.post() with header Authorization: Bearer WorkspaceClient().config.token - Fetch country/global averages by querying the Delta table via WorkspaceClient().sql (or spark if available) - Handle connection errors gracefully with user-friendly messages Tech stack: Gradio for UI, Plotly for charts, databricks-sdk for table queries, requests for endpoint. Generate: app.py, requirements.txt, app.yaml, and any helper files needed.
What Claude Code will generate
  • app.py — full Gradio app with inputs, outputs, REST call to endpoint
  • requirements.txt — gradio, plotly, databricks-sdk, requests, pandas
  • Possibly a helpers.py for Delta table queries and SDK auth
Tips for iterating with Claude Code:
1
Test locally after generation
Say "run the app locally to test it" — Claude Code will install deps and start the server. Open the URL it prints.
2
Fix issues with exact error text
Paste the full error message from the terminal. Don't paraphrase — the exact error text is the best context Claude Code can receive.
3
Iterate in small steps
"the bar chart labels are missing, fix that" · "the country dropdown shows too many options, add a search" · One fix at a time.
4
Review before accepting
Claude Code shows you diffs before applying. Read them — especially for the endpoint URL and token variable name.
No encoding needed in the app
The model is a sklearn Pipeline — OrdinalEncoder is baked in. Pass "Maize" and "Argentina" directly in the payload. If Claude Code tries to encode before calling the endpoint, tell it: "The serving endpoint handles all preprocessing — send raw strings."

Deploy to Databricks
⏱ 15 min
4

Once the app works locally, deploy it with a single command. Databricks Apps handles packaging, routing, and auth automatically.

Windows (PowerShell)
# Deploy your app — uses the gdm-workshop OAuth profile (no token needed) databricks apps deploy --name gdm-yield-app-<your_username> --profile gdm-workshop
Mac / Linux
# Deploy your app — uses the gdm-workshop OAuth profile (no token needed) databricks apps deploy --name gdm-yield-app-<your_username> --profile gdm-workshop
Expected output
  • CLI shows progress: packaging → deploying → ready
  • App URL printed: https://<workspace>.azuredatabricks.net/apps/gdm-yield-app-<username>
  • Open the URL in your browser — app loads with SSO, no login required
  • Enter inputs → click Predict → see the yield prediction and charts
How authentication works
Deploy uses the gdm-workshop OAuth profile configured in Step 2 — no PAT token needed. At runtime, Databricks Apps handles SSO for browser access. The app itself authenticates to the SDK and Model Serving endpoint via WorkspaceClient(profile="gdm-workshop"). You don't write any auth code.
If deploy fails
Check logs: databricks apps logs --name gdm-yield-app-<your_username> — the error message will show what's wrong (usually a missing dependency in requirements.txt or a syntax error in app.py). Paste the error into Claude Code and say "fix this deploy error".

Shiny vs. Databricks Apps — The Full Picture
⏱ 5 min
5

You just rebuilt a typical Shiny workflow end-to-end on Databricks. Here's what changed — and what stayed the same.

What you do in R / Shiny What just happened
Write ui.R + server.R (200+ lines) Described the app to Claude Code → code generated
renderPlot({ ggplot(...) }) gr.Plot() with interactive Plotly chart
reactive({ model$predict(input) }) REST call to Model Serving endpoint (independently scaled)
Bundle model inside the app as .rds Model runs independently, versioned in Unity Catalog, monitored
rsconnect::deployApp() databricks apps deploy — one command
R package management (renv) requirements.txt
~20 concurrent users max (free tier) Serverless — auto-scales, no server to manage
Update model = redeploy app Update model endpoint = app automatically uses new version
The key architectural shift
In Shiny, the app IS the model. In Databricks Apps, the app is just the UI — the model is a separate REST service. This means: update your model without touching the UI, scale the model and app independently, and track every model version in Unity Catalog.
